## 11.3 深入设计

- 深入探讨 feed 的发布流程
  - Web Server
    - 用户身份校验 和 限流
  - 广播服务(broadcast service)
    - 广播模型
      - 写广播(推送模型)
        - 预先计算将推送帖子给哪些用户好友
        - 优势
          - 实时生成, 立即推送
        - 缺点
          - Hotkey problem(热键问题), 用户好友过多时, 查询所有用户生成news feed 耗时上升
          - 不活跃用户, 浪费计算资源
      - 读广播(拉取模型)
        - 读取的时候才生成, 按需模型
        - 优点
          - 节省计算资源, 只在用户访问时才生成
          - 避免不活跃用户浪费资源
          - 不会有 热键问题, 没有主动推送操作
        - 缺点
          - 没有预先计算, 获取 news feed 很慢
      - 实际使用常用
        - 混合上述两种策略
        - 图数据库管理好友列表
        - 消息队列推送数据
        - 数据存储在缓存中
- 深入探讨 news feed 的获取流程
- 缓存架构
  - news feed层：
    - 存储了news feed的ID。
  - 内容层：
    - 存储每条帖子的数据。流行的内容存储在热点内容缓存中。
  - 社交图谱层：
    - 存储用户关系数据。
  - 操作层：
    - 存储用户对帖子点赞、回复，或者进行其他操作的信息。
  - 计数器层：
    - 存储点赞、回复、关注等行为的计数数据。
