# 设计一致性哈希系统

> 实现横向扩展, 均匀在服务器之间均匀分配请求

## 5.1 重新哈希的问题

- 常见负载平衡算法
  - 服务器序号 = hash(key)%N
    - N: 服务器数量
    - 受服务器数量影响严重
      - 数量变动后, 容易导致大量缓存未命中(Cache Miss)

## 5.2 一致性哈希

> 一致性哈希是一种特殊的哈希。如果一个哈希表被调整了大小，那么使用一致性哈希，则平均只需要重新映射k/n个键，这里k是键的数量，n是槽(Slot)的数量

- 利用哈希空间和哈希环
  - 确定映射顺序(顺时针), 将 key 存储到最近 服务器上
- 每次新加(移除)服务器, 只用调整少部分 key 重新分配位置
  - key 存储原则: 离自己最近的服务器
- 基本步骤
  - 使用均匀分布的哈希函数将服务器和键映射到哈希环上。•
  - 要找出某个键被映射到了哪个服务器上，就从这个键的位置开始顺时针查找，直到找到哈希环上的第一个服务器
- 虚拟节点
  - 尽可能让key均匀分布在所有服务器上, 服务器的hash不一定是均匀分布在 哈希换上
    - 给每台服务器用多个虚拟节点表示.
      - e.g. 原始server1 ---> 虚拟节点后: ser1_0, ser1_1, ser1_2 ...
      - 使得每个服务器都对应多个分区段
- 找到受影响的键
  - 需要重新分配的键
  - 新加服务器节点后, 逆时针(分配规则是 顺时针移动)移动找到已存在服务器
    - 这个 哈希段内的key就是本次改动后受影响的 key 范围, 需要重新计算归属服务器

## 5.3 总结

- 好处
  - 添加或者移除服务器的时候，需要重新分配的键最少。
  - 更容易横向扩展，因为数据分布得更均匀。
  - 减轻了热点键问题, 过多访问一个特定分区可能会导致服务器过载
