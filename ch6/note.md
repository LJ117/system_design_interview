# 设计键值存储系统

> 键值存储也叫键值数据库, NoSQL数据库, 是一种非关系型数据库, 以键值对的形式存储数据。

- 键值对
  - 每一个唯一的标识符作为键（Key），与之对应的值（Value）可以是任意类型的数据。
  - 处于性能考虑: 较短的键更好
    - 纯文本键, 哈希键

## 6.1 理解问题并确定设计边界

- 权衡考量
  - 读写操作, 内存使用
- 特点
  - 键值对大小限定
  - 可存储大数据
  - 高可用性
  - 高可扩展
  - 自动伸缩
  - 可调节一致性
  - 低延时

## 6.2 单服务器的键值存储

- 优化
  - 数据压缩
  - 仅保存频繁使用数据在内存

## 6.3 分布式键值存储

- CAP理论
  - 分布式系统可达成理论
    - 最多只能 3选2
  - 一致性(Consistency)
    - 所有客户端在相同时间点看到数据是同样的
  - 可用性(Availability)
    - 有节点故障时, 系统仍旧可用
  - 分区容错性(Partition Tolerance)
    - 两个节点间通信中断(网络不可达), 系统仍可以继续运行
  - 实际应用
    - CP（一致性和分区容错性）系统：
      - 支持一致性和分区容错性，但牺牲了可用性。
    - AP（可用性和分区容错性）系统：
      - 支持可用性和分区容错性，但牺牲了一致性。
    - CA（一致性和可用性）系统：
      - 支持一致性和可用性，但牺牲了分区容错性。
      - 因为网络故障是无法避免的，所以分布式系统必须容忍网络分区。
      - 因此，在现实世界中CA系统不可能存在。
- 数据分区
  - 大量全部数据集不可能全部放置于单一服务器上
    - 最实用场景是 分割为小的分区, 存储在多台服务器上
    - 注意
      - 均匀分布
      - 节点变动时, 尽量减少数据迁移
        - 一致性哈希技术
- 数据复制
  - 数据在N台服务器上异步复制, 同步数据
    - 会使用虚拟节点技术(尽可能保证 均匀复制数据到不同 服务器上)
      - 虚拟节点, 要跳过相同服务器
- 一致性
  - 数据在不同副本间的复制和同步
  - 仲裁一致性(Quorum Consensus)可以保证读写操作的一致性。
    - Quorum:
      - 参与一个读或写操作的最小节点数或副本数
  - 一致性模型
    - 强一致性模型：
      - 任何读操作返回的值都是最新写入的数据。客户端永远不会看到过时的数据。
    - 弱一致性模型：
      - 随后的读操作返回的可能不是最新的值。
    - 最终一致性模型：(**推荐选择**)
      - 这是弱一致性的一种特殊形态。
      - 经过足够长的时间，所有的数据更新都会传播开来，并且所有副本会变得一致。
- 不一致性解决方案: 版本控制
  - 每次修改数据都生成一个新的不可变的数据版本
- 故障处理
  - 故障检测
    - 至少有两个独立信息源才能标记服务器故障
    - 去中心化故障检测方法
      - Gossip 协议
        - 每个节点维护一个节点成员列表, 包含成员ID和心跳计数
        - 每个节点定期增加自己的心跳书
        - 每个节点定期给一组随机节点发送心跳信号, 这些节点又将心跳信号继续传递给另一组节点
        - 节点收到心跳信号, 更新自己的成员列表
        - 心跳在预定时间内没有增加, 则该成员被判定宕机
    - 处理临时故障
      - 松散仲裁(Sloppy Quorum)
        - 发现服务器故障后, 用另一个服务器替代接受请求
        - 服务器恢复后, 重新接受请求
    - 处理永久故障
      - 实现了反熵协议(Anti-entropy Protocol)来保持副本同步
      - 使用 哈希(Merkle)树比较
    - 处理数据中心故障
      - 多数据中心间同步数据
