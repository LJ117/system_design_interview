# 系统设计面试的框架

## 3.1 有效的系统设计的四个步骤

### 3.1.1 第一步: 理解问题并确定设计边界

> 问正确的问题, 做合适的假设, 并收集构建系统所需信息

- 问题是否能得到回答
  - 能
  - 不能
    - 写下自己的假设
- 常见问题
  - 需要构建什么样的具体功能?
    - 用户端呈现
      - Web? 移动APP?
  - 产品用户规模
  - 公司预计扩展系统时间?预计3个月, 6个月 和 1年后的系统规模如何?
  - 当前技术栈是什么? 有哪些现有服务可以简化设计?
    - 是否有可复用组件
- 针对产品特性, 用户规模, 使用细节针对性提问

### 3.1.2 第二步：提议高层级的设计并获得认同

- 为设计制定一个初始蓝图
  - 征集面试官反馈, 作为自己队友, 一起参与讨论
- 在白板或纸上画出关键组件框图
  - 客户端(APP,web端)
  - API
  - web服务器
  - DB
  - CDN
  - 消息队列 ...
- 做封底估算
  - 评估设计是否满足系统需求
  - 表明自己的思考过程

### 3.1.3 第三步：设计继续深入

- 规划架构组件的优先级顺序
- 注意时间管理
  - 尽量展示自己的能比, 避免不必要细节讨论

### 3.1.4 总结

- 系统设计不可能完美
  - 一直有改进的机会
- 复述设计方案的有用性
- 故障场景分析讨论
  - 服务器故障
  - 数据丢包
- 运维问题
- 规模性扩展
  - 100w --> 1000w
- 正确操作
  - 总是向对方寻求明确的解释, 不依赖自己的假设
  - 理解问题的要求
  - 没有最佳解法
  - 让面试官知道你的想法, 持续保持沟通
  - 如果有可能给出多个方案
  - 设计蓝图达成一致后, 深入讨论细节. 先设计最重要的组件
  - 试探对方想法, 好的面试官是队友一起解决问题
  - 永不言弃
- 禁忌
  - 对常见问题不熟悉
  - 没有弄清需求和假设就给出方案
  - 一开始就讨论细节, 永远先给出高层设计
  - 思路卡住的时候, 自己干着急, 不寻求帮助
  - 拒绝沟通, 一个人自己思考
  - 给出方案后要等面试官说结束, 尽早尽量频繁的征求反馈

## 3.2 面试中每一步的时间分配

1. 第一步：理解问题并确定设计的边界（3～10分钟）​。
2. 第二步：提议高层级的设计并获得认同（10～15分钟）​。
3. 第三步：设计继续深入（10～25分钟）​。
4. 第四步：总结（3～5分钟）​。
