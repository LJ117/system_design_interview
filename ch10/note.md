# 设计通知系统

> 可以向用户发送重要信息

- 类型
  - 手机推送通知
  - 短信
  - 邮件

## 10.1 理解并确定设计边界

- 确认设计目标
  - 通知类型
  - 实时非实时?
  - 支持设备
  - 触发方式
    - 主动 or 被动?
  - 是否支持取消订阅
  - 通知规模

## 10.2 高层级设计

- 不同类型的通知
  - IOS
    - 服务商 -> APNS(苹果通知推送服务) -> 用户设备
  - Android
    - 服务商 -> FCM(/firebase cloud messaging) -> 用户设备
  - 短信
    - 一般采用第三方提供的短信服务API
  - 邮件
    - 可自己设置邮件服务器
    - 也可以选择商业邮件服务
- 联系信息的收集流程
  - 用户安装或注册时收集用户联系信息, 持久化存储到数据库中
- 通知的发送/接收流程
  - 高层级设计
    - 服务1到服务N
      - 微服务, 定时任务
      - 支持不同类型, 不同阶段的信息选择不同通知方式
      - 提供不同API满足不同服务的通知需求
    - 通知系统
      - 核心组件
      - 内部身份认证流程
      - 消息队列处理
    - 第三方服务
      - 注意可扩展性
      - 特定区域选用支持该地区的通知组件

## 10.3 深入设计

- 可靠性
  - 数据防丢
    - DB持久化通知机制 + 重试机制
      - 重试需要设计 **去重(dedupe)机制**
- 其他组件和要考虑因素
  - 通知模板
    - 预定义好的消息模板, 开发使用只用填写固定变量即可
  - 通知设置
    - 用户可设置通知的接收方式, 以及是否接收某些类型的通知
  - 流量控制
    - 避免用户被"轰炸", 主动设置通知频率
  - 重试机制
    - 引入消息队列重新发送
      - 增加告警策略, 通知开发人员
      - 监控队列消费情况, 如果有积压需要调整策略
  - 事件追踪
  - 系统监控
- 更新后设计

## 10.4 总结

- 可靠性
- 安全性
- 追踪和监控
- 尊重用户设置
  - 根据用户设置是否选择接受通知
- 流量限制
