# 从0到100万用户的扩展

## 1.1 单服务器配置

- 常见所有应用, 缓存(Redis), 数据库DB 都放置在该单台机器上

## 1.2 数据库

- 伴随业务,用户基数的增长. 需要多台服务器性能支撑
  - 一台专用业务流量
  - 一台用作数据库
  - 方便分别进行扩展
- 数据库分类
  - 关系型数据库(RDBMS)
    - MySQL, Oracle, Postgres
  - 非关系型数据库(NoSQL)
    - Neo4j, HBase, MongoDB
    - 一般不支持连接(join)操作

## 1.3 纵向扩展 vs 横向扩展

- 纵向扩展
  - 提升单台服务器性能
    - CPU,RAM etc
    - 适用流量小场景
    - 硬件性能限制
    - 无故障转移能力和冗余
- 横向扩展
  - 向外扩展, 添加多台服务器
  - 大型应用适用

## 1.4 负载均衡器

> 将输入流量均匀分配到负载均衡集里的各个Web服务器中(横向扩展)

- 用户流量达到公共IP的负载均衡器上
  - 用户不会直接访问具体Web服务器
  - **服务器** 间, **服务器** 和 **负载混衡器** 间 通信使用私有IP地址

## 1.5 数据库复制

- 主从数据库模式(常见: 一主多从)
  - 原始数据库(Master 主库)
    - 通常只支持 **写操作**(插入, 更新, 删除)
  - 拷贝数据库(Slave 从库)
    - 保存数据副本且仅支持 **读操作**
- 优势
  - 性能更好
    - 读写分离, 能支撑更多 并行处理查询操作
  - 可靠性高
    - 从数据库可升级主数据库
    - (若唯一)从库宕机, 主库可支持 **读操作**
  - 可用性高

## 1.6 缓存

> 存储响应耗时, 内存热点数据
> 常见缓解 直接查询 数据库 压力

- 使用场景
  - 频繁读, 少写
- 过期策略
  - 一定要有过期, 自动删除时间
- 单点故障(SPOF: signle point of failure)
  - 但缓存服务器会遇到
  - 部署多个数据中心避免单点故障
- 驱逐策略
  - 当缓存已满, 新增环境会删除已有条目
  - 常见策略
    - LRU
    - LFU
    - FIFO

## 1.7 内容分发网络(CDN)

> 地理位置上分散的服务器网络, 用于传输静态内容
> 缓存: 图片, 视频, CSS 和 Js文件等 静态内容

- 注意事项
  - 花销
    - 仅存储常用内容
  - 合理的缓存过期时间
  - CDN回退
    - CDN故障时, 保证服务不中断
  - 作废文件
    - 在文件过期前清理文件
      - 手动调用 API 清理
      - 以最新版本替换老版本, 版本号控制

## 1.8 无状态网络层

> 横向扩展, HTTP 本身无状态
> 会话数据在持久性存储中(DB), Web服务器都通过数据库访问状态数据

- 有状态架构
  - 服务器存储用户状态和数据
- 无状态架构
  - Web服务器仅作业务逻辑处理, 数据从数据存储中拉取
  - 系统更健壮, 更容易扩展
  - 常见共享存储
    - RDBMS
    - NoSQL(Redis, Memcached)
      - 易扩展

## 1.9 数据中心

> 用户基于地理信息的域名自动分流(geoDNS)
> 负载均衡器负责具体导流工作

- 必须解决技术
  - 流量重定向
  - 数据同步
  - 测试部署

## 1.10 消息队列

> 这是一个持久化的组件, 存储在内存, 支持异步通信
> 用作缓冲区,分配异步请求

- 基本架构
  - 输入服务(生产者)
  - 消费者
    - 生产者暂时不可用, 消费者仍能继续从队列中读取信息
    - 现有LLM处理, 大量异步处理适用

## 1.11 记录日志, 收集指标与自动化

- 记录日志
  - 监控错误日志
  - 三方原始请求和响应
    - 用于排错
  - 汇总日志到中心化服务, 方便搜索和查看
- 收集指标
  - 不同类型的指标数据
    - 用于商业洞察和分析
    - 系统健康检测
      - 主机
        - CPU, MEM, Disk I/O
      - 聚合
        - 整个数据库性能, 缓存性能
      - 关键业务
        - DAU, 留存, 收益统计
- 自动化
  - CI CD

## 1.12 数据库扩展

- 纵向扩展
  - 为已有机器增加算力
    - CPU, Mem, Disk
    - 单机硬件性能邮上限
    - 单机容易有单点故障
    - 总成本偏高
- 横向扩展(分片)
  - 添加更多服务器
  - 常用哈希函数来分配数据存储使用DB
  - 分片策略
    - 分片键(Sharding key), 分区键(Partition Key)
      - 标准
        - 让数据均匀分布
    - 重分片数据
      - 数据增长过快, 单Shard不再满足需求
      - 数据分配不均
        - 大量数据分配到单一Shard上命中
      - 名人问题(Hot key)
        - 需要把这种热点访问, 主动分开存储到不同Shard, 降低单Shard频繁访问导致性能下降
      - 连接和去规范化
        - 分片后, 无法直接使用 join 查询
        - 数据冗余存储一份

## 1.13 超过百万用户量

- 总结
  - 让网络层无状态。
  - 每一层都要有冗余。
  - 尽量多缓存数据。
  - 支持多个数据中心。
  - 用CDN来承载静态资源。
  - 通过分片来扩展数据层。
  - 把不同架构层分成不同的服务。
  - 监控你的系统并使用自动化工具。
