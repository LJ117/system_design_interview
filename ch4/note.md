# 设计限流器

> 限流器: 控制客户端或服务端发送的流量速率
> HTTP的世界中, 限流器限制的是客户端在一定时间内被允许发送的请求数量

- 好处
  - 防御 拒绝服务攻击(Denial of Service Dos)
    - DDOS 分布式拒绝服务攻击(distributed denial-of-service attack)
  - 降低成本
    - 减少请求, 减少使用付费的第三方API
  - 预防服务器过载
    - 降低服务器负载

## 4.1 第一步：理解问题并确定设计的边界

> 收集整理 真实需求, 不同算法有其不同优劣

## 4.2 第二步：提议高层级的设计并获得认同

### 4.2.1 在哪里实现

- 客户端
  - 一般而言不是强制实行限流的可靠位置
  - 容易被恶意伪造
  - 无法控制客户端行为
- 服务端
- 中间件
  - API网关组件

### 4.2.2 流量限制算法

- 代币桶算法(Token Bucket)
  - 简单易用, 内存使用率高
  - 预定义容量, 按照固定速率放入 可消耗 Token, 放满为止
  - 每次请求消耗一个 Token
- 漏桶算法(Leaking Bucket)
  - 请求先存在桶中, 如果桶已满则丢弃, 否则追加任务队列中
  - 定期从任务队列取出请求处理
    - 关键参数
      - 桶大小(队列大小), 总任务数
      - 出栈速度, 每秒可处理请求数, 通常每秒为时间单位
  - 缺点
    - 突发流量会造成大量积压, 导致新请求被限流抛弃
- 固定窗口计数器算法(Fixed Window Counter)
  - 每个时间窗口给一个处理阈值(一个时间段内只能处理这么多请求)
    - 到达阈值后, 当前窗口内的请求全抛弃
  - 优
    - 内存使用效率高
    - 容易理解
    - 每单位时间结束重置请求阈值, 适用特定场景
  - 劣
    - 窗口边界流量激增, 迅速达到阈值 请求被丢弃
- 滑动窗口日志算法(Sliding Window Log)
  - 原理
    - 记录请求时间戳(存入Cache)
    - 每次新请求来到, 移除当前时间窗口起始时间前的请求时间戳
    - 每次新请求记入日志时与当前设定阈值对比
      - 超过阈值, 拒绝请求
  - 优
    - 流量限制精确, 永远不会超过阈值
  - 劣
    - 消耗内存过多, 内次都要计算, 更新窗口
      - 已被拒绝的请求, 请求时间戳也被保留在内存中
- 滑动窗口计数器算法(Sliding Window Counter)
  - 原理
    - 固定窗口+滑动日志
    - 其中一种公式:
      - 滑动窗口所允许的请求数量 = 当前窗口的请求数+之前窗口的请求数×滑动窗口和之前窗口的重合率
        - 场景:
          - 每分钟(窗口)允许 7个请求, 前一分钟有5个, 当前分钟有3个. 当一个新请求出现在当前分钟的30%的位置时
            - 3 + 5 \* (1 - 30%) = 6.5
            - 3 + 5 \* 0.7 = 6.5(实际使用会取(向上或向下)取整)
  - 优
    - 平滑流量波动
    - 内存使用高效
  - 劣
    - 对不够严格的回溯窗口生效
    - 对真实流量做近似估计, 不够精确

### 4.2.3 高层级架构

> 需要计数器针对同一用户(用户指纹), 限制用户请求

- 计数器存储位置
  - 内存存储, 速度满足
    - 常见: Redis
      - INCR: 原子加
      - EXPIRE: 过期时间设置

## 4.3 第三步：设计继续深入

- 流量限制规则
  - 在配置文件写限制规则(保存在硬盘上,持久化存储)
- 超过流量限制
  - HTTP 响应 429(请求过多)
    - 如果是订单, 应入队保存, 之后再处理
    - 响应头
      - 当前时间窗口剩余可用数量
      - 单位窗口可请求总数
      - 限流多久后回复
- 详细设计
  - 流量限制规则持久化保存再硬盘上
  - 客户端请求首先发送到 **限流器中间件** 上
  - **限流器中间件** 从 **缓存** 上加载规则
- 分布式系统中的限流器
  - 挑战
    - 竞争条件(Race Condition)
      - 高并发场景面临问题
      - 解决方案
        - 加锁
        - Lua脚本和Redis有序集合数据结构
    - 同步
      - 多个限流器处理流量
      - 解决方案
        - 黏性会话(Sticky Session)
        - 中性化存储
- 性能优化
  - 多数据中心
  - 最终一致性模型
- 监控
  - 确保 算法, 规则 有效

## 4.4 总结

- 多层限制
  - 不仅是 HTTP 应用层, 也可以考虑 网络层(IP层)
- 最佳实践
  - 使用客户端缓存，避免频繁地调用API。
  - 理解流量限制，不要在短时间内发送太多请求。
  - 添加代码以捕获异常或错误，使客户端可以优雅地从异常中恢复。
  - 在重试逻辑中添加足够的退避时间。
